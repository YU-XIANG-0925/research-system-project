# 階段 0：研究與探索

**目標**：分析功能規格，以識別未知數、風險和外部依賴性。此階段著重於在規劃「如何做」之前，澄清「做什麼」和「為什麼做」。

**輸入**：`specs/002-andriod-app-robot/spec.md`

## 1. 需求分析

- **FR-001**：用於功能選擇的主頁面。
- **FR-002**：用於 MQTT 代理的連線頁面。
- **FR-003**：持續的 MQTT 連線。
- **FR-004**：「播放機器人動作」頁面，包含動作列表。
- **FR-005**：在「播放機器人動作」頁面以 每秒一次 的頻率發布馬達角度。
- **FR-006**：「手動機器人調整」頁面。
- **FR-007**：在「手動機器人調整」頁面將馬達角度初始化為零。
- **FR-008**：在「手動機器人調整」頁面以 每秒一次 的頻率顯示馬達角度。
- **FR-009**：「記錄」按鈕，用於發布馬達角度。
- **FR-010**：「返回」按鈕，用於將馬達角度重設為零。
- **FR-011**：專案位於 `@robot` 目錄中。
- **FR-012**：基於 Android 的解決方案。

## 2. 釐清 (向利害關係人提問)

- **[需要釐清]** 發布到 MQTT 的馬達角度資料的預期格式是什麼？
- **[需要釐清]** 當機器人未回應指令時，應用程式應如何處理？
- **[需要釐清]** 「官方預定義的機器人動作」的具體細節是什麼？

## 3. 風險評估

- **高風險**：確保在不同應用程式功能之間建立穩定且持續的 MQTT 連線。
- **中風險**：即時資料同步和顯示 (每秒一次 更新)。
- **低風險**：UI 佈局和導覽。

## 4. 外部依賴性

- **NuwaSDKExample**：專案必須嚴格遵循此範例專案的結構和實作。
- **MQTT 函式庫**：需要一個基於 Java 的 MQTT 用戶端函式庫。
- **機器人 SDK**：位於 `tools` 目錄中的專有 SDK。
