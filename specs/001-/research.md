# 演講能力輔助系統研究報告

## 講稿檔案格式 (FR-001)
- **決策**：初期支援 `.txt` 格式。
- **理由**：純文字最容易解析與處理。對 `.docx` 或 `.pdf` 的支援可作為後續增強功能。
- **考慮過的替代方案**：`.docx` (需要 `python-docx` 函式庫)、`.pdf` (需要 `PyPDF2` 函式庫)。這些方案會增加初期版本不必要的複雜性。

## 匯出格式 (FR-004)
- **決策**：匯出為包含講稿與標註的 JSON 檔案。
- **理由**：JSON 是一種結構化格式，對機器和人類都易於解析。它可以輕鬆地用於將已標註的講稿重新匯入系統。
- **考慮過的替代方案**：匯出回原始格式。這將難以實作，特別是對於像 `.docx` 或 `.pdf` 這樣的格式。

## 影片分析與手動調整 (FR-006)
- **決策**：對於手動調整，提供一個簡單的介面來命名動作，並將其與要透過 MQTT 發送的命令關聯。對於影片分析，使用 `mediapipe` 來偵測姿勢，並允許使用者將特定姿勢儲存為自定義動作。
- **理由**：這在使用者控制與自動化之間取得了平衡。手動調整簡單直接。使用 `mediapipe` 進行影片分析是一個強大的功能，可以透過現有函式庫來實現。
- **考慮過的替代方案**：更複雜的影片分析技術，這將需要更多的研究與開發時間。

## 連線參數 (FR-010)
- **決策**：系統將需要 MQTT 代理 (broker) 的 IP 位址、埠號以及用於發布命令的主題 (topic)。
- **理由**：這些是 MQTT 通訊所需的標準參數。
- **考慮過的替代方案**：其他通訊協定，如 WebSockets，但 MQTT 更適合物聯網裝置通訊。
